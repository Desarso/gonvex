# Current Limits

Gonvex is under active development. These docs describe the intended product shape and the local prototype that is being built toward it.

## Implemented Direction

The current codebase proves these pieces locally:

- app-local `gonvex/` folder
- Go schema declarations
- generated TypeScript API references
- local runtime process
- WebSocket subscriptions
- mutation calls
- Postgres-backed task data
- LiveGrid-style visible-window subscriptions
- HTTP gzip and WebSocket compression
- local package build/watch workflow

## Not Fully Generic Yet

Function execution is still becoming generic. The runtime needs a general dispatcher that can execute arbitrary registered app functions without project-specific paths.

## Current CLI Boundary

The npm CLI currently implements:

```bash
npm create gonvex@latest my-app
npx gonvex init
npx gonvex login
npx gonvex project create my-app
npx gonvex token create "Developer CLI"
npx gonvex dev
npx gonvex env list
npx gonvex env push .env.production
```

`tenant`, `deploy`, `generate`, and `doctor` remain roadmap commands. See the
[CLI Reference](./cli-reference) for current authentication and project-provisioning
flags.

## Multi-Project Operations Are Early

The runtime and dashboard have development project registry endpoints, project keys,
environment variables, and per-project manifests. The operational model is not yet a
hardened hosted control plane.

Needed work:

- allowed origins and runtime auth
- environment/deployment records
- dashboard UI for personal-token lifecycle and more granular custom roles
- production upgrade and rollback workflows

## Multi-Tenant Runtime Is Roadmap

The target model supports users who belong to multiple tenants with roles and permissions. Tenants may route to separate databases.

Needed work:

- tenant registry
- user registry
- memberships
- roles and permissions
- active tenant switching
- tenant database routing
- tenant-aware cache and invalidation scoping

## Invalidation Is Conservative

The current invalidation strategy favors visible correctness over minimal refreshes.

Future work should track query dependencies generically:

- tables read
- predicates
- ranges
- sort keys
- returned row IDs
- changed columns
- tenant database route

## Schema Migrations Are Early

Schema sync is appropriate for local development and safe data-preserving changes. The runtime should reject schema changes that would break existing rows and print the error in the CLI.

Production migrations need:

- migration versioning
- previews
- explicit destructive-change confirmation
- rollback strategy
- per-tenant rollout control
- large-table backfill strategy

## Auth Is Early

Dashboard sessions, account personal access tokens, project memberships, and scoped
project/token permissions are implemented. OAuth/device authorization, organization
roles, recovery flows, and hosted-control-plane hardening are still incomplete.

Do not expose development data-browser endpoints as application APIs.

## Storage Is Early

File storage is optional. Apps that need files should attach an S3-compatible provider; the local MinIO service in this repo is only an example for development. Upload URL flows, tenant scoping, and storage authorization policy still need design.

## Production Guidance

Treat Gonvex as an active prototype until these areas are complete:

- generic function execution
- project and tenant routing
- auth and permissions
- production migrations
- deployment/versioning
- operational dashboard workflows
